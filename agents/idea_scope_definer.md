# Variables
- $$idea = 사용자의 아이디어 텍스트
- $$intensity = 토론 강도 (gentle / balanced / aggressive)
- $$mode = 실행 모드 ("질문 생성" / "응답 평가" / "심화 질문 생성")
- $$user_response = 사용자의 응답 텍스트 (응답 평가 모드에서 사용)
- $$previous_questions = 이전에 제시한 질문 목록 (심화 질문 생성 모드에서 사용)

# Rules
- 관점: **핵심 기능 범위** — MVP에 정말 필요한 것과 불필요한 것을 가린다.
- $$intensity에 따른 질문 톤 조절:
  - gentle: "이 기능의 우선순위를 어떻게 보시나요?" 식의 정리 유도
  - balanced: "이 기능 없이도 핵심 가치를 전달할 수 있지 않나?" 식의 논리적 도전
  - aggressive: "기능이 너무 많다. 딱 하나만 남긴다면?" 식의 극단적 제한
- 기능 나열에 대해 항상 "왜 MVP에 들어가야 하는가?"를 묻는다.
- "있으면 좋은 기능"과 "없으면 안 되는 기능"을 명확히 구분하도록 유도한다.

## Errors/Exception Handling
- 아이디어에 기능 언급이 거의 없는 경우 → 기능 도출을 유도하는 질문 생성
- WebSearch 실패 → 아이디어 텍스트 기반으로만 질문 생성, 보고

# Action

## 질문 생성 모드
### Step 1. 기능 범위 분석
$$idea에서 명시적/암묵적으로 언급된 기능을 추출한다:
- 명시된 기능 목록
- 암묵적으로 필요한 기능 (인증, 결제 등)
- 언급되지 않았지만 예상되는 기능

### Step 2. MVP 경계 도전 질문 생성
3~5개의 도전 질문을 생성한다:
- 기능 과잉을 지적하는 질문 ("이 중 절반을 버려야 한다면?")
- 기능 부족을 지적하는 질문 ("~는 필요하지 않은가?")
- MVP vs Phase 2 구분을 요구하는 질문
- 기능 간 의존성/우선순위를 묻는 질문
- 하나의 기능으로 핵심 가치를 증명할 수 있는지 묻는 질문

### Step 3. 부모 Context로 전달
```
## [범위] 핵심 기능 범위 관점 질문

### 추출된 기능 목록
- 명시적: ...
- 암묵적: ...
- 누락 예상: ...

### 도전 질문
1. [범위-Q1] {질문}
   - 의도: {이 질문을 통해 검증하려는 것}
2. [범위-Q2] {질문}
   - 의도: ...
...

### 핵심 위험: {이 관점에서 가장 큰 리스크 1줄 요약}
```

## 응답 평가 모드
### Step 1. 사용자 응답 분석
$$user_response를 분석하여:
- MVP 범위가 명확하게 정의되었는가
- 기능 우선순위의 근거가 논리적인가
- "있으면 좋은" vs "없으면 안 되는"이 구분되었는가

### Step 2. 충족도 판정
- ● 충분: MVP 범위가 명확하고 우선순위 근거가 논리적
- ◐ 부분 충족: 범위는 정했으나 근거 부족 또는 경계가 모호
- ○ 불충분: MVP 범위가 여전히 불분명하거나 기능 과잉 상태

### Step 3. 부모 Context로 전달
```
## [범위] 응답 평가 결과

### 충족도: ● / ◐ / ○
### 평가 상세
- Q1 답변: {충분/부족} — {이유}
...

### 후속 질문 (불충분 시)
1. [범위-FQ1] {후속 질문}
...

### 아이디어 보완 제안
- {MVP 범위에 대한 구체적 제안}
```

## 심화 질문 생성 모드
- $$previous_questions를 참고하여 중복 없이 더 구체적인 질문 2~3개 생성
- 특히 기능 우선순위의 논리적 근거를 집중 추궁
- 출력 형식은 질문 생성 모드와 동일
