# Goal
이 파일을 읽은 Agent는 소스코드를 분석하여 핵심 구성 요소를 식별하고, 그에 맞는 디렉토리와 마크다운 파일을 동적으로 생성한다.
# Examples
$$architecture = ./examples/nest-architecture-example.md
$$index = ./examples/INDEX-exmple.md
# Structure
```
docs/
└── guides/
    ├── architecture/
    │   └── architecture.md   # 전체 레이어 구조 개요
    ├── convention/
    │   └── {구성요소}.md               # 소스코드 분석 후 동적 결정 (하단 참고)
    ├── workflow/
    │   └── {작업명}.md                 # 반복 레이어 작업 절차 (하단 참고)
    └── INDEX.md                        # 모든 문서 링크 취합
```
# 작성 규칙
## 생성 순서
- 1단계. 소스코드 전체 탐색: 구성 요소 목록 및 아키텍쳐 구조 확정
  - 의존성 디렉토리(ex. .venv, node_modules 등) 제외
- 2단계. convention/*.md 생성(병렬 생성 가능)
- 3단계. architecture/{architecture_name}.md (convention 파일 목록 확정 후)
- 4단계. workflow/*.md (architecture에서 반복 레이어 구조가 발견된 경우)
- 5단계. INDEX.md (모든 파일 작성 완료후 링크 취함)

## Convention/ 구성요소 동적 결정 규칙
### Step1. 소스코드에서 핵심 구성 요소 식별
- Agent는 주요 레이어와 모듈을 식별하여 구성 요소를 결정한다.
- 구성 요소는 API 레이어, 데이터베이스, 인프라, 인증, 인가, 캐싱, 메시징, 모니터링, 로깅, 배포, 테스트, 문서화 등으로 분류된다.
- 각 구성 요소는 마크다운 파일로 문서화된다.
### Step2. 파일명 결정 규칙
- 구성 요소의 책임단위로 파일을 생성한다.
- 하나의 경로에 여러 책임이 섞여있으면 책임별로 분리한다.
  - ex. guards/ 내에 jwt와 permissions이 공존 -> guards/jwt.md, guards/permissions.md
- 유사한 구성 요소는 하나의 파일로 통합한다.
  - ex. create.service/find.service/update.service -> service.md
### Step3. 각 *.md 파일의 내용 구성 규칙
```
# {구성요소명} 컨벤션

## 개요
<!-- 이 구성 요소의 책임과 위치 -->

## 구조
<!-- 소스코드 기반 실제 파일 경로 및 클래스/함수 목록 -->

## 사용 패턴
<!-- 실제 코드에서 어떻게 사용되는지 예시 -->

## 주의사항
<!-- 이 구성 요소 사용 시 반드시 알아야 할 제약 -->
```
- "사용 패턴" 섹션에서 가능하면 Good/Bad 또는 Before/After 대비 코드를 포함한다.
- "구조" 섹션에서 ASCII 다이어그램 외에 mermaid 다이어그램도 허용한다.
- 설정/빌드 구성요소는 파일 간 의존 관계 다이어그램과
  주요 워크플로우(컴파일/개발/빌드)를 포함한다.

## architecture/{architecture_name}.md 작성 규칙
> example. $$architecture를 참고하여, 아키텍처 md 파일 작성
### Step1. 아키텍처 파악
> 주요 질문 흐름에 따라 소스코드를 분석한다.
- 요청이 어떻게 진입하는가?
  - 앱 진입점(시작 파일)에서 바인딩된 포트, 프로토콜, 미들웨어 등 확인
  - 앱 초기화시 무엇을 설정하는가?
- 진입 후 어떤 단계를 거치는가?
  - 전역 등록된 미들웨어(Guard, Interceptor, Pipe, Filter 등) 목록과 실행 순서 확인
- 레이어가 몇 개이고 각각의 책임은 무엇인가?
  - 요청이 통과하는 API의 체인을 추적(ex. Controller -> Service -> Repository 등)
- 응답은 어떤 방식으로 포맷되는가?
  - 공통 응답 wrapper 클래스, 함수 등의 존재 여부 확인
- 예외 처리는 어떤 방식으로 진행하는가?
  - 사용자 정의 예외 존재여부를 파악
  - 예외 처리 흐름 파악
- 서버 전반에 대한 모니터링 여부 파악
- 기타 사항 파악
### Step2. project Overview
> 프로젝트 전체적 디렉토리 구조를 작성.
### Step3. 다이어그램 작성
> 다이어그램은 아키텍처의 실제 흐름을 반영. 형식은 흐름 유형에 따라 선택.
- 단방향 레이어 흐름
```
┌──────────────────────┐
│  {entry point}       │
└──────────┬───────────┘
           │
┌──────────▼───────────┐
│  {next layenr}       │
│  - {component}       │
└──────────────────────┘
```
- 분기 흐름
```
           ┌───────────────────┐
           │  {decision point} │
           └────┬──────────────┘
        ┌───────┴───────┐
┌───────▼──────┐  ┌─────▼────────┐
│  {Path A}    │  │  {Path B}    │
└──────────────┘  └──────────────┘
```
- 박스 안에는 실제 클래스명·미들웨어명 사용 (추상 명칭 금지)
- 소스코드에 존재하지 않는 단계는 다이어그램에 포함하지 않음
- 흐름이 복잡하면 전체 흐름 다이어그램 1개 + 부분 상세 다이어그램으로 분리
- 실행 환경이 2개 이상(예: Electron/Browser, Server/Serverless)이면,
  환경별 통신 흐름을 별도 서브섹션으로 작성한다.
### Step4. 레이어별 개요와 구성요소 .md 파일 링크 작성
ex. Layer1

| 문서 | 내용 |
|-----|------|
| [Component1](../convention/component1.md) | 구성요소1 |
| [Component2](../convention/component2.md) | 구성요소2 |

### Step5. 주의사항 작성

## workflow/ 작성 규칙
### 생성 조건
- architecture 분석 결과, 반복되는 레이어 구조가 발견될 때 생성한다.
### 파일명 결정 규칙
- 작업 단위를 kebab-case로 명명한다. (예: `add-new-domain.md`)
### 내용 구성 규칙
- 레이어 역순으로 작업 단계를 구성한다.
- 각 단계마다 코드 템플릿을 포함한다.
```
# {작업명} 워크플로우

## Step 1. {최하위 레이어}
<!-- 코드 템플릿 -->

## Step 2. {상위 레이어}
<!-- 코드 템플릿 -->

## Step N. {최상위 레이어}
<!-- 코드 템플릿 -->
```

## INDEX.md
> docs/guides/INDEX.md 는 Agent가 프로젝트를 파악하기위한 문서 진입점.
> example $$index 를 참고하여 INDEX.md를 작성하시오.
- INDEX.md는 모든 문서의 링크를 모아둔 파일로, 각 구성 요소의 마크다운 파일에 대한 링크를 포함한다.
- INDEX.md는 구성 요소별로 분류되어 있으며, 각 구성 요소의 링크는 해당 구성 요소의 마크다운 파일로 연결된다.
- INDEX.md는 문서의 구조와 내용을 쉽게 찾을 수 있도록 도와준다.
### Step1. 정보 수집
INDEX.md는 모든 문서가 완성된 후 마지막에 작성.
- architecture/*.md
- convention/*.md 파일들
### Step2. 섹션 작성
아래는 판단 기준이며 고정 목록이 아님. 생성 근거가 있는 경우에 작성

| 섹션 | 작성 조건 | 내용 기준 |
|-------|-------------|-------------|
| 개요 | 항상 작성 | 이 디렉토리가 문서화하는 프로젝트,범위를 한 문단으로 기술 |
| Convention | 항상 작성 | 실제 생성 파일 트리, 각 파일 옆 한 줄 설명 주석 |
| 아키텍처 | architecture/*.md 존재 | 파일명, 설명 2열 테이블, 각 파일로의 상대 경로 링크 |
| 컨벤션 | convention/*.md 존재 | 책임 유형별 소제목으로 그룹화, 파일명,설명, 관련 경로 테이블 |
| 프로젝트 고유 성질 | 도메인 전반에 반복 패턴이 있는 경우 | 하단 Step 3  참고 |
| Workflow | workflow/*.md 존재 | 파일명, 설명 2열 테이블, 각 파일로의 상대 경로 링크 |
| 코드 리뷰 체크리스트 | convention에서 필수 규칙이 3개 이상 | convention 문서의 필수 규칙을 체크박스로 취합 |
### Step3. 프로젝트 고유 요약 테이블 결정 원칙
INDEX.md의 핵심은 링크 모음을 넘어 프로젝트를 처음 보는 사람이 전체 구조를 빠르게 파악할 수 있게 하는 것입니다.
소스코드에서 도메인 전반에 반복되는 패턴을 발견하면, 그것을 요약 테이블로 추가합니다.

| 발견 패턴 | 테이블로 만들 것 |
| -------- | ---------------|
| 모든 도메인이 동일 레이어 구조를 따른다 | 레이어별 역할, Base 클래스 요약 |
| 파일명에 일관된 명명 규칙 존재 | 레이어별 파일명 패턴/예시 |
| 공통 데코레이터,어노테이션 조합 반복 | 데코레이터 사용처/설명 요약 |
| 그 외 프로젝트 고유 반복 규칙 | 해당 규칙에 맞는 테이블 자유 형식 작성 |
