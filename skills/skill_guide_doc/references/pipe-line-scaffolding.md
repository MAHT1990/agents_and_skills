# Goal
이 파일을 읽은 Agent는 소스코드를 분석하여 특정 프로세스(부트스트랩, 요청 처리, 빌드 등)의 실행 파이프라인을 단계별로 추적하고, 그에 맞는 디렉토리와 마크다운 파일을 동적으로 생성한다.

# Examples
$$pipe-line-example = ./examples/bootstrap-pipeline-example.md

# Structure
```
docs/
└── guides/
    └── pipe-line/
        └── {pipe-line-name}/           # 파이프라인별 디렉토리 (분석 대상에 따라 동적 결정)
            ├── 00-overview.md          # 전체 파이프라인 다이어그램 + 문서 구성
            ├── {NN}-{step-name}.md     # 소스코드 분석 후 동적 결정 (하단 참고)
            └── ...
```

### 디렉토리 이름 결정 규칙 ({pipe-line-name})
- 분석 대상 프로세스를 kebab-case로 명명
- 예시:

| 분석 대상 | 디렉토리명 |
|----------|-----------|
| 앱 부트스트랩 과정 | `bootstrap` |
| HTTP 요청 처리 과정 | `request-lifecycle` |
| WebSocket 연결 과정 | `websocket-connection` |
| 빌드/번들 과정 | `build-pipeline` |
| CI/CD 배포 과정 | `deploy-pipeline` |
| 인증 흐름 | `auth-flow` |

- 동일 소스코드에 여러 파이프라인 분석이 가능하므로, 이름으로 구분한다.

# 작성 규칙

## 전제 조건
- pipe-line 문서는 기존 guide docs(architecture/, convention/)가 존재할 때 **보완 문서**로 작성된다.
- 기존 guide docs가 없으면, 먼저 해당 scope의 scaffolding(api-server/client)으로 guide docs를 생성한 뒤 pipe-line을 작성한다.
- 기존 guide docs가 있으면, convention/*.md 파일을 참조 컨텍스트로 활용한다.

## 생성 순서
- 1단계. **분석 대상 파이프라인 식별**: 사용자가 명시한 프로세스의 진입점을 소스코드에서 찾는다.
- 2단계. **파이프라인 이름 결정**: 분석 대상에 따라 `{pipe-line-name}` 디렉토리명을 결정하고, 사용자에게 확인받는다.
- 3단계. **메서드 호출 체인 추적**: 진입점부터 종료점까지 메서드 호출을 재귀적으로 추적하여 전체 단계를 나열한다.
- 4단계. **단계 그룹화**: 추적된 호출 체인을 논리적 단계로 그룹화하고, 각 단계의 경계를 결정한다.
- 5단계. **00-overview.md 생성**: 전체 파이프라인 다이어그램 + 단계별 문서 링크 테이블
- 6단계. **{NN}-{step-name}.md 생성**: 각 단계를 개별 문서로 작성 (병렬 생성 가능)
- 7단계. **INDEX.md 업데이트**: 기존 INDEX.md에 pipe-line/{pipe-line-name} 섹션 추가

## 단계 그룹화 규칙

### Step1. 진입점 → 종료점 전체 호출 체인 나열
- 소스코드에서 실제 메서드 호출을 추적한다.
- 추상 명칭이 아닌 **실제 클래스명·메서드명**을 사용한다.
- 각 호출에 대해 **파일:라인번호** 참조를 기록한다.

### Step2. 논리적 단계 경계 결정
아래 기준으로 호출 체인을 단계별로 분리한다:

| 분리 기준 | 설명 | 예시 |
|----------|------|------|
| 책임 전환 | 담당 클래스/모듈이 바뀌는 지점 | NestFactory → DependenciesScanner |
| 비동기 경계 | await로 대기하는 주요 작업 단위 | scan() 완료 후 createInstances() |
| 사이드이펙트 경계 | 외부 상태(컨테이너, 파일, 네트워크)가 변경되는 지점 | Container에 모듈 등록 완료 |
| 조건 분기점 | 중요한 if/switch 분기가 있는 지점 | Singleton vs Request-scoped 처리 |
| 사용자 인식 단위 | 프레임워크 사용자가 인지하는 개념 단위 | "미들웨어 등록", "라우트 등록" |

### Step3. 파일 번호·이름 결정 규칙
- 접두사: `{NN}-` (00부터 시작, 2자리 zero-padding)
- 00은 항상 overview
- 이름: 해당 단계의 핵심 클래스 또는 개념을 kebab-case로
  - ex. `01-nest-factory.md`, `03-scanner.md`, `08-middleware.md`
- 하나의 단계가 너무 크면(500줄 초과 예상) 세부 단계로 분리
  - ex. `04-instance-loader.md` + `05-injector.md` (InstanceLoader의 세부 과정)

## 00-overview.md 작성 규칙

### 필수 섹션

```
# {파이프라인명} 개요

## 목적
<!-- 이 파이프라인이 무엇을 하는지 1~2문장 -->

## 전체 파이프라인 다이어그램
<!-- ASCII 다이어그램: 진입점 → 종료점 전체 흐름 -->
<!-- 들여쓰기로 하위 단계 표현 -->
<!-- 각 단계에 [N] 번호 표기 -->

## 문서 구성
<!-- 테이블: | 문서 | 단계 | 내용 | -->

## 주요 소스코드 위치 참조
<!-- 테이블: | 파일 | 핵심 위치 | -->
```

### 다이어그램 형식
```
EntryPoint()
│
├─ [1] {단계명}
│       ├─ {하위작업1}
│       └─ {하위작업2}
│
├─ [2] {단계명}
│       └─ {하위작업}
│
└─ [N] {최종단계}
```

- 실제 메서드명·클래스명 사용 (추상 명칭 금지)
- 화살표(←) 주석으로 핵심 동작 설명
- 조건 분기는 들여쓰기 + 조건문으로 표현

## {NN}-{step-name}.md 작성 규칙

### 필수 섹션

```
# Step {N}. {단계명}

## 개요
<!-- 이 단계의 책임과 위치를 1~2문장으로 -->

## {핵심메서드명}() 시그니처
<!-- 파일 경로, 라인 번호 -->
<!-- 파라미터와 반환값 명시 -->

## 처리 흐름
<!-- 메서드 호출 체인을 의사코드 형태로 -->
<!-- 각 단계에 소스코드 참조(파일:라인) 포함 -->

## 조건 분기 (해당 시)
<!-- 테이블: | 상황 | 처리 | -->

## 주의사항
<!-- 이 단계에서 반드시 알아야 할 제약·위험 -->
```

### 처리 흐름 작성 형식
```
MethodName(param1, param2)
  │
  ├─ 1. {작업 설명}
  │     → {호출되는 메서드}()   (파일:라인)
  │
  ├─ 2. 조건 분기
  │     ├─ 조건A → {처리}
  │     └─ 조건B → {처리}
  │
  └─ 3. {작업 설명}
        → {반환값 또는 사이드이펙트}
```

### 코드 참조 규칙
- 모든 메서드 호출에 **파일:라인번호** 참조를 포함한다.
  - 형식: `packages/core/scanner.ts:86`
  - 라인 번호가 불확실하면 `:` 뒤에 근사값을 적되 `~` 접두사: `packages/core/scanner.ts:~90`
- 주요 메서드는 시그니처(파라미터 타입, 반환 타입)를 명시한다.
- 상수/토큰은 정의 위치를 참조한다.

### 조건 분기 테이블 형식
```
| 상황 | 처리 |
|------|------|
| 조건A | 처리A |
| 조건B | 처리B |
| 기본값 | 처리C |
```

## INDEX.md 업데이트 규칙
- 기존 INDEX.md의 디렉토리 구조에 `pipe-line/{pipe-line-name}/` 트리 추가
- 아키텍처 섹션 아래에 **"Pipeline 분석"** 섹션 추가 (이미 존재하면 하위에 추가)
- 동일 프로젝트에 여러 파이프라인이 있을 수 있으므로 파이프라인별 소제목으로 구분

### INDEX.md 추가 섹션 형식
```markdown
## Pipeline 분석

### {파이프라인 한글명}

| 문서 | 단계 | 내용 |
|------|------|------|
| [00-overview.md](./pipe-line/{name}/00-overview.md) | 전체 | 전체 파이프라인 다이어그램 |
| [01-{step}.md](./pipe-line/{name}/01-{step}.md) | 1 | {내용} |
| ... | ... | ... |
```

# 분석 깊이 기준

## 반드시 추적해야 하는 것
- 메서드 호출 체인 (진입점 → 종료점)
- 주요 조건 분기 (if/switch)
- 비동기 경계 (await 지점)
- 사이드이펙트 (외부 상태 변경: DB, 컨테이너, 파일시스템)
- 에러 처리 경로 (try-catch, 예외 전파)

## 추적하지 않아도 되는 것
- 유틸리티 함수의 내부 구현 (결과만 기술)
- 타입 변환/캐스팅 세부사항
- 로깅·디버깅 코드
- deprecated 경로
- 테스트 전용 코드

## 분석 깊이 판단 기준

| 깊이 | 기준 | 예시 |
|------|------|------|
| 상세 추적 | 파이프라인의 주 경로(main path) | Scanner.scan() → scanForModules() → scanModulesForDependencies() |
| 요약 기술 | 주 경로에서 호출되는 보조 메서드 | calculateModulesDistance() → "BFS로 distance 설정" |
| 생략 가능 | 파이프라인과 무관한 부수 기능 | 로깅, 메트릭 수집 |

# 제약 사항
- 소스코드에 존재하지 않는 단계를 추측하여 포함하지 않는다.
- 다이어그램과 처리 흐름에는 실제 클래스명·메서드명만 사용한다.
- 파일:라인 참조가 불확실하면 생략하지 말고 `~` 접두사로 근사값을 기록한다.
- 하나의 단계 파일이 500줄을 초과할 것으로 예상되면 세부 단계로 분리한다.
