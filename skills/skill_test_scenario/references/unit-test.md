# Unit Test Scenario 작성 가이드

## 목적
함수/모듈 단위로 독립적인 동작을 검증하는 테스트 시나리오를 도출한다.

## 분석 범위 ($$unit_scope)
### diff 모드
- `git diff` 결과에서 변경된 파일/함수만 분석 대상으로 한다.
- 변경된 함수의 입력/출력/부수효과를 중심으로 TC 도출.

### full 모드
- $$src_path 하위 전체 소스코드를 분석 대상으로 한다.
- 의존성 디렉토리(node_modules, .venv 등) 제외.

## 분석 기준
### Step1. 테스트 대상 식별
- export된 함수/클래스/메서드 목록 추출
- 각 함수의 책임(역할) 파악
- 순수 함수 vs 부수효과 함수 구분

### Step2. TC 도출 기준
각 함수에 대해 아래 케이스를 고려:

| 유형 | 설명 | 예시 |
|---|---|---|
| Happy Path | 정상 입력 → 정상 출력 | 유효한 인자로 호출 |
| Boundary | 경계값 테스트 | 0, 빈 문자열, 빈 배열, MAX_INT |
| Negative | 잘못된 입력 | null, undefined, 타입 불일치 |
| Exception | 예외 발생 확인 | throw 여부, 에러 메시지 |
| Return Value | 반환값 타입/구조 검증 | 객체 필드, 배열 길이 |

### Step3. TC 형식
```
[TC-{번호}] {시나리오 제목}
- 대상: {함수명/클래스명.메서드명}
- 유형: Happy Path / Boundary / Negative / Exception
- 입력값: ...
- 테스트 절차: ...
- 예상 결과: ...
```

## 주의사항
- 외부 의존성(DB, API, 파일시스템)은 mock 대상으로 표기
- private 함수는 직접 테스트하지 않고, public 인터페이스를 통해 간접 검증
- 테스트 1개당 검증 포인트는 1개로 제한 (단일 책임)
